## What this lap did (Brick 5c — `wip/PathCOmega.lean`, axiom-clean)

Lap 104's 2nd in-kernel finding said the `imax` stored ordinal is INSUFFICIENT for the cut node's
operator-control (the `cut` constructor needs `sord premise ≺ stored`, but the max-ACHIEVING premise EQUALS
`imax`, never `≺` — `icmp` irreflexive), and framed the fix as Gentzen's rank-aware `ω`-tower, deferred as
"genuinely multi-month". **This lap shows that deferral is unnecessary for the principal ∀/∃ step.** The
natural (Hessenberg) sum `inadd` (`#`) — already in the repo (`InternalNadd.lean`, used by the induction
node's `indOmegaStoredOrd`) — supplies BOTH the operator-control AND the descent obligations at once, in
five new axiom-clean lemmas:

- **`lt_inadd_self_right`** `: isNF X → isNF g → icmp 0 g = 0 → icmp X (inadd X g) = 0` — `X ≺ X # g` for
  `g ≻ 0`. The operator-control fact `imax` could NOT provide (strict self-domination on the right summand).
  One-liner from `inadd_left_mono` + `inadd_zero_right`.
- **`lt_inadd_self_left`** `: isNF X → isNF g → icmp 0 X = 0 → icmp g (inadd X g) = 0` — `g ≺ X # g` for
  `X ≻ 0`. From `inadd_right_mono` + `inadd_zero_left`.
- **`inadd_strict_mono`** `: a≺a' → b≺b' → a#b ≺ a'#b'` (all NF) — `#` strictly monotone in BOTH arguments.
  `icmp_trans'` over `inadd_right_mono` then `inadd_left_mono`.
- **`redAllExN`** — the `#`-stored ∀/∃-cut reduct: identical to lap-103's `redAllEx` but the stored ordinal
  is `(sord selected-∀-premise) # (sord ∃-premise)`, not their `imax`. It sums the reduced premises' OWN
  `sord` (not `iord`), correct even when a premise is itself a cut/ω-node (general Path-C), unlike `imax`'s
  `iord` left field.
- **`zcOK_redAllExN`** — **principal ∀/∃-cut `hinv`, FULL closure, `imax`-free.** The `#`-stored reduct of a
  `ZcOK` cut (left = ω-∀-node, right = ∃-node) is `ZcOK`. **Operator-control is DISCHARGED** from the
  premises' positivity + NF alone (`lt_inadd_self_right`/`_left`) — NO externally-supplied `hLctrl`/`hRctrl`.
  Contrast lap-104's `zcOK_redAllEx_of_ctrl`, which had to ASSUME `hLctrl`/`hRctrl` and could not prove them
  for `imax`. This closes the operator-control half of `hinv` for the principal ∀/∃ step.
- **`sord_redAllExN_lt`** — the `#`-stored reduct STRICTLY drops the stored ordinal **against a `#`-stored
  parent**: if the reduct's two premises each have `sord ≺` the parent's corresponding premise ordinals
  (`αAll`, `αEx`), then `(sord …) # (sord …) ≺ αAll # αEx` (the parent's own `#`-stored ordinal), by
  `inadd_strict_mono`. **No additive-principality of the parent ordinal needed** — which is the obstruction
  that drove lap 104 to `imax`. The parent's `#`-stored ordinal is itself the operator-controlled value, and
  strict-monotonicity carries the descent. So BOTH operator-control AND descent hold for the SAME `#` choice.

### Why this is the right resolution (not a shortcut)

This is exactly the standard Schütte `#`-bookkeeping (Towsner's META cut-elimination in `src/Zinfty.lean`
combines cut premises by natural sum). Lap 104's `imax`-vs-`inadd` tension was real but the conclusion ("need
the tower NOW") over-shot: `imax` gives descent-vs-ARBITRARY-`α` but no operator-control; `inadd` gives both,
because a properly operator-controlled parent stores `#` of ITS premises, so the descent is against a
`#`-value and strict-monotonicity (not principality) carries it. The reduct is NOT principal so its own
ordinal need not be principal — only strictly below the parent's, which `#`-monotonicity delivers.
