reducts are built exactly that way (`isChainInf_of_last`, j₀ = lh ds − 1), but `isChainInf` only
PROMISES `∃ j₀ < lh ds`, so a redex could in principle sit in the un-threaded tail `(j₀, lh ds)` whose
cut rank is unbounded. Confirmed dead ends this lap:
- The library pairing is **not** coordinate-monotone under `≤` (`⟪0,big⟫` vs `⟪1,0⟫`), so
  `redexCode ≤ ⟪i0,j1⟫` (inference pair, `j1 ≤ j0`) does NOT give `redexI ≤ i0` or `redexJ ≤ j1`.

  `A_i = chainAsucc ds redexI` (R-side) and `A_i ∈ chainAnt ds redexJ`, `A_i ∉ seqAnt s` (L-side) —
  but the **threading** that would pin `A_i = chainAsucc ds i'` (i' < redexJ ≤ j₀, hence rank ≤ r)
  only holds for `i ≤ j₀`. So the proof bottlenecks on exactly `redexJ ≤ j₀`.
